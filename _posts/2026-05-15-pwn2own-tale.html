Pwn2Own is the premier desktop-and-browser exploitation contest: vendors ship their latest builds, researchers show up with zero-days, and whoever chains the most bugs in the shortest time takes home the prize — and the laptop.

This post is a sanitized version of our team's run at the most recent edition. We'll cover the bugs we used, the chain we built, and the lessons we learned. No unpatched vulnerabilities are discussed.

The target

We targeted a major browser with a two-bug chain: a renderer sandbox escape, followed by a kernel-level privilege escalation. The renderer bug was a use-after-free in the JavaScript engine; the kernel bug was a logic error in the IPC broker.

Bug one: renderer escape

The UAF lived in the JIT compiler's handling of a specific Array.prototype pattern. By carefully shaping the heap, we could get the JIT to emit code that dereferenced a freed object, giving us a controlled type confusion and, eventually, arbitrary read/write in the renderer process. 

// Simplified trigger — not the real exploit
let a = new Array(0x100);
let b = a.map(x => ({x}));
// After GC, one of the objects is freed but still referenced
// by a stale pointer in the JIT'd code.

Bug two: sandbox escape

With arbitrary R/W in the renderer, we needed to break out. The browser's sandbox relies on a broker process that validates IPC messages. We found that one of the broker's message handlers didn't check the size of an embedded buffer, allowing a heap overflow in the broker itself — which runs with medium integrity.

Putting it together

  1. Serve a page that triggers the UAF.
  2. Use the type confusion to leak a WebAssembly module's backing store.
  3. Overwrite a vtable pointer to gain code execution in the renderer.
  4. From the renderer, send a malformed IPC message to the broker.
  5. Overflow the broker's heap, hijack control flow, and run shellcode.

Total time from first test to reliable shell: four hours. Most of that was spent on heap shaping — the bugs themselves were straightforward once we understood the code paths.

Lessons learned

  • Read the source first. We spent two days reading the IPC broker before we found the bug. The actual exploit took an afternoon.
  • Automate heap shaping. Manual heap feng shui is slow. Write a script that tries thousands of layouts.
  • Test on a clean image. Contest environments are locked down. Test on a vanilla VM first.

Pwn2Own is a reminder that even the best-engineered software has seams. The goal isn't to shame vendors — it's to make exploitation harder and detection easier, one bug at a time.
