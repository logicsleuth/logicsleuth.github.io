Cryptography is hard to get right — not because the math is difficult, but because the implementation details are unforgiving. A single off-by-one, a reused nonce, or a missing MAC can undo years of peer-reviewed security.

This post catalogs the mistakes I see most often in real codebases. None of these are theoretical; they've all shown up in production systems I've audited.

1. Reused nonces with AES-GCM

AES-GCM is an AEAD cipher: it provides confidentiality and integrity. But it depends on a unique nonce for every encryption under the same key. Reuse a nonce, and an attacker can recover the authentication key and forge messages.

The most common cause: using a counter that resets on restart, or generating nonces with a PRNG that has a short period. The fix is simple: use a 96-bit random nonce (collision probability is negligible) or a monotonic counter that survives reboots.

2. ECB mode

Electronic Codebook mode encrypts each block independently. Identical plaintext blocks produce identical ciphertext blocks. The classic demonstration is the "ECB penguin" — an image encrypted with ECB is still recognizable.

Never use ECB. Use CBC, CTR, or (preferably) an AEAD mode like GCM or ChaCha20-Poly1305.

3. Missing authentication

Encryption without authentication is a recipe for disaster. If you encrypt with AES-CBC but don't add a MAC, an attacker can flip bits in the ciphertext and make predictable changes to the decrypted plaintext. This is the basis of padding-oracle attacks.

The rule: encrypt-then-MAC. Encrypt the plaintext, then MAC the ciphertext. Or better, use an AEAD mode that does both.

4. Hardcoded keys

I've seen encryption keys hardcoded in source code, compiled into binaries, and stored in plaintext config files. The moment the source or binary leaks, the encryption is useless.

Use a key management system (KMS) or, at minimum, load keys from environment variables or a secrets manager. Rotate keys regularly.

5. Weak random number generation

Math.random() in JavaScript, rand() in C, java.util.Random — none of these are cryptographically secure. If you're generating nonces, keys, or tokens, you need a CSPRNG: crypto.getRandomValues(), java.security.SecureRandom, or os.urandom().

6. Timing side-channels

A naive string comparison (==) short-circuits on the first differing byte. An attacker can measure the response time to learn how many bytes of a secret they've guessed correctly. The fix: use a constant-time comparison function.

The meta-lesson

Don't roll your own crypto. Use a high-level library (libsodium, Tink, WebCrypto) that makes the right thing the easy thing. If you must use a lower-level primitive, read the documentation twice and have someone else review your code.
